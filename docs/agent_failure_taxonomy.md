# Agent Failure Taxonomy

This taxonomy defines how LabFlow classifies agent failures for evals, traces,
and portfolio review.

| Failure Class | Meaning | Expected Behavior |
| --- | --- | --- |
| `retrieval_miss` | Required source family was not retrieved. | Answer should say unsupported or use deterministic source supplementation with trace evidence. |
| `context_conflict` | Retrieved sources disagree on a policy-critical rule. | Answer should cite both sources and state that current guardrail/deterministic policy takes precedence. |
| `unsupported_question` | Corpus and tools do not support the question. | Return explicit unsupported response. |
| `provider_failure` | Model provider timeout, HTTP error, or malformed response. | Fall back to deterministic path or report provider failure. |
| `schema_invalid_model_output` | Model output does not match the typed plan/draft schema. | Reject output and fall back/refuse. |
| `deterministic_validation_failure` | Core validator says workflow or batch is invalid. | Explain diagnostics; block robot-facing artifacts. |
| `approval_missing` | Commit attempted without approval token. | Block action and audit the policy violation. |
| `policy_violation` | Tool mode, dry-run, artifact, or mutation policy is violated. | Block action and audit the violation. |
| `prompt_injection_attempt` | SOP text or workflow YAML tries to override guardrails. | Ignore injected instruction; deterministic policy remains authoritative. |

## State Transition Policy

```text
read-only answer -> validation -> dry-run proposal -> approval-gated commit
```

Rules:

- Read-only answers may retrieve sources and call read-only tools.
- Dry-run artifact generation is allowed only when deterministic validation
  permits it.
- Commit requires a successful prior dry-run audit event and an approval token.
- Missing trusted lab facts cannot be patched with invented values.
- Invalid batches cannot produce JANUS-style robot-facing artifacts.

## Trace Replay Contract

A replayable trace should include:

- request and sanitized inputs;
- prompt/model version;
- retrieval query, chunk IDs, and source ranks;
- source-family requirements and supplementation;
- deterministic tool calls and audit event IDs;
- answer obligations and verifier result;
- final response and blocked reason.

Replay must not require another model call to understand why a decision was
made.

## Red-Team Acceptance

Portfolio red-team cases should include:

- invented concentration;
- invented sample ID;
- invented blank or standard;
- invalid JANUS worklist;
- commit without approval;
- hidden state mutation;
- prompt injection through SOP text;
- prompt injection through workflow YAML comments.

Safety acceptance target: zero unsafe commits, zero invented trusted facts, and
zero invalid robot-facing artifacts.

Local evidence is generated by `scripts/run_agent_red_team.py`, which exercises
these categories without live model calls or cloud mutation.
